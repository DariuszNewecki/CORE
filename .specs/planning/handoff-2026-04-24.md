## The investigation — Option A was a false lead

2026-04-22's handoff named this session's lead as "Option A: IntentGuard CLI-depth rule scope narrowing" on the theory that `cli.resource_first` (scope declared as `src/cli/resources/**/*.py`) was somehow firing against `src/will/workers/violation_remediator.py` during the `fix.placeholders` churn. The proposed fix was a YAML scope narrowing in `.intent/enforcement/mappings/`.

Three diagnostic passes ruled this out. **On-disk state:** `cli.resource_first` is defined once per registry, with the correct scope; the error string appears nowhere under `src/`; the pre-`cf2ea63c` `action_results` rows persist only the flat `error_message` string, so the `PolicyRule` that actually matched is not recoverable from the database. **Out-of-process load:** a freshly-instantiated IntentGuard against `/opt/dev/CORE` reports `is_valid=True` for the target under both `impact="write-code"` and `impact=None`. Every `cli.*` rule returns `Path.match=False` against the target. **In-daemon live dump** (the critical one): one IntentGuard singleton, 140 rules loaded, all `cli.*` scopes intact, `check_transaction` clean, two FileHandler instances both correctly pointing at the singleton. No rule mutation. No instance duplication. No scope drift.

The timestamps settled it. Failure window was 2026-04-23 14:34:58 – 16:46:05 UTC (nine failures, ~10-minute cadence). Self-recovery at 16:55:34 UTC (proposal `63d47846…`, commit `15c8ae3e`) against the same file with the same action. Current daemon started at 19:05 UTC, ~2 hours after the self-recovery. The process that produced the failures no longer exists. In-memory state at failure time is unrecoverable from any angle.

The conclusion the evidence forces: the failure was real, transient, self-recovered without a code change, and its cause is not recoverable from the current system. Narrowing the YAML scope would either have had no effect (if the cause is not scope-related) or would have masked a structural bug behind a scope override. Either outcome breaks the enforcement-real layer. ADR-009 closes the lead honestly.

## The in-daemon introspection capability

Yesterday's "look inside the running daemon" was not a capability we had. It is now. The procedure:

1. `sudo apt install gdb` (one-time; standard dev tool).
2. Payload script under `reports/diagnostics/` (read-only, introspection only — checked into git).
3. Invocation: `sudo /opt/dev/CORE/.venv/bin/python -c "…gdb one-liner using -ex per command…"` that attaches to the daemon PID, calls `PyGILState_Ensure`, executes the payload, releases the GIL, detaches. Full procedure captured in the session transcript for reuse.

Notes and gotchas that bit during the session, kept here so the next instance doesn't repeat them:

- `pyrasite 2.0` is unreliable. It shells out to `gdb` without checking the return code; if `gdb` isn't installed or the attach fails, it prints "injected OK" anyway. Silent-fail.
- `pyrasite` prints `"WARNING: ptrace is disabled. Injection will not work."` on Ubuntu Yama systems regardless of `ptrace_scope` value (it reads scope at import time, not at inject time). This warning appears even on `--help`. Ignore it if using `sudo gdb` directly.
- `ptrace_scope=1` is the default on this kernel and **is not overridden by `CAP_SYS_PTRACE`** via sudo — the scope check runs before privilege evaluation. For direct `sudo gdb` attach, this is fine (gdb bypasses pyrasite's scope check). For `pyrasite` as root, it silently no-ops.
- The one-shot `sysctl kernel.yama.ptrace_scope=0 → pyrasite → sysctl=1` flap was used once in this session and is noted as a recoverable but undesirable posture change. It is not required for the `sudo gdb` direct-attach path.

Test it before needing it. The payload is at `reports/diagnostics/_ig_live_payload.py`; it writes a markdown report to `reports/diagnostics/2026-04-24_intentguard_live_dump.md`. Re-running is cheap.

## Carry-over — next session

**Lead item: Option B — Revival contract verification.**

**The contract.** The Finding paper (`.specs/papers/CORE-Finding.md`, section 7a) specifies:

> When a Proposal reaches `failed` status, the ProposalConsumerWorker MUST:
> 1. Query all Findings whose payload `proposal_id` matches the failed Proposal's ID.
> 2. For each such Finding, set `status = 'open'`, `claimed_by = NULL`, `claimed_at = NULL`.
> 3. Post a `report` entry recording the revival: the Proposal ID, the count of revived Findings, and the failure reason.

The Remediation paper (`.specs/papers/CORE-Remediation.md`, line 130085 in the 2026-04-22 export) names ProposalConsumerWorker as the responsible actor and cross-references the Finding paper for the full contract.

**The code.** `src/will/workers/proposal_consumer_worker.py::ProposalConsumerWorker.run()` handles proposal failure by: incrementing `failed` counter, logging `"ProposalConsumerWorker: proposal '%s' failed — %s"`, appending to `results` list, posting a summary `report` at run end. It does not query findings. It does not reset any finding status. It does not post a revival report.

`src/will/autonomy/proposal_state_manager.py::ProposalStateManager.mark_failed()` updates only the `autonomous_proposals` table (status + execution_completed_at + failure_reason + execution_results). It does not reach into findings.

Grep across `src/` for `revive`/`revival`/`reopen.*finding`: zero hits. All occurrences are in `.specs/papers/` constitutional text. The contract is specified but not implemented.

**The empirical signature matches.** 2026-04-22's handoff observed that the `fix.placeholders` churn posts a new finding for the same subject on each iteration rather than reopening the prior one. The DB shape ("two resolved rows at different timestamps for the same subject") is exactly what "revival not wired" would produce: the sensor detects the same violation → posts a new finding → gets deduped? — no, it doesn't, because the prior finding went to `resolved` (via `deferred_to_proposal` pathway) rather than staying open. So the sensor re-posts. Not a bug in the sensor; a gap in the proposal-failure path.

**Proposed shape for Option B.** Investigation first, then implement. Two options for where the work lands:

1. **Inside `ProposalConsumerWorker.run()`**, in the `else` branch where `failed += 1`. Calls into a new method (likely on BlackboardService) that implements the three-step contract atomically. This is the paper's literal assignment — "the ProposalConsumerWorker MUST."
2. **Inside `ProposalStateManager.mark_failed()`**, as a side-effect after the UPDATE to `autonomous_proposals`. Keeps the revival coupled to the state transition rather than to the worker. Arguably more robust — any future code path that marks a proposal failed (there are other call sites, e.g., the test-requirement-failure branch in ProposalExecutor) would trigger revival without needing to duplicate the logic.

Option 2 is likely the right answer on the "coupled to state transition, not to caller" principle, but that's a call for the session, not a pre-decided. Either way the implementation pattern is: the three steps in section 7a, executed in one SQL transaction, with the revival report posted after commit.

**Sensor deduplication concern worth verifying early.** Paper says "Revived Findings re-enter the remediation loop as if freshly posted. The deduplication contract applies — if the same violation still exists, the sensor will not re-post it; the revived Finding is sufficient." If the sensor's dedup logic looks only at open findings (not at historically-resolved ones), revival needs to either flip the historical row or post a new open finding. Reading the sensor code is the pre-check before the fix.

**Session shape.** Shaped like the 2026-04-22 Option B (engine-contract widening): a contract-wiring verification with a specific answer. Probe-verify rhythm applies — read `ProposalConsumerWorker.run()`, read `ProposalStateManager.mark_failed()`, read whichever sensor posts findings for `fix.placeholders` (likely PurityObserver or similar), confirm the dedup logic, then produce the complete corrected file.

## Hazards worth naming

**In-memory governance state is not externally observable.** Today's live dump was the first time CORE's runtime rule state had been looked at from outside. The capability now exists but is manual, requires sudo, and only works while the process still exists. Today's failure window (~2 hours, ~9 failures, self-recovered) would have been invisible if not for the pre-`cf2ea63c` rows in `action_results`. A failure that self-recovers silently without leaving DB evidence would leave no trace at all.

**Pre-`cf2ea63c` evidence is lost.** All nine failing rows from 2026-04-23 carry only the flat `error_message`. The matching `PolicyRule` is not recoverable. If the revival contract fix (Option B) reveals that findings from that window were not revived correctly, reconciliation is not mechanical — there's no `proposal_id → finding_id` trace for the pre-`cf2ea63c` rows beyond what's in `autonomous_proposals.scope.files`.

**The cold-registry hazard from 2026-04-22 is still live.** Daemon reload semantics for action decorators remain as they were — stale until restart, no signal, no warning. Nothing this session addressed that.

**Today's daemon has not reproduced the CLI-depth block.** That is not evidence the bug is fixed; it's evidence the bug is silent. If the failure mode is boot-time transient, the current clean state survives only until the next restart. Any `core-admin` change that requires a daemon restart reopens the exposure window. Worth watching in the next few restarts.
