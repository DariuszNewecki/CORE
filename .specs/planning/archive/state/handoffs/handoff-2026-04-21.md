## Carry-over — next session

Three candidates, listed by leverage.

### Option A — Fix Audit Ingest Worker resolution path
Single worker, single method probably. Clearest blocker to ever
trusting the daemon's output: right now stuck claims accumulate
on every sensor pass. Without this, the daemon "works" but the
blackboard becomes a growing debt pile, not a converging work
queue. Convergence Principle violation, directly.

### Option B — Fix `details: {}` stripping
The audit report is lying — sensors emit rich structural signal,
the JSON strips it. Every downstream consumer of `details` is
blind. Once fixed, dashboards and the vectorization layer can
surface the class-structure signal ADR-007 produces; without the
fix, ADR-007's `dominant_class_*` fields exist only in sensor
memory and nowhere else.

### Option C — ADR-007 addendum + author ADR for the 13-new-findings review
Consequences section correction, then a per-file review ADR (or
catalogue) naming what to do with each of the 13 surfaced files —
domain-warranted, extract collaborators, compose, or subpackage.

### Recommendation

**A before B.** Resolution-path bug is Convergence-Principle-blocking;
it corrupts every future observation of the daemon. B is also urgent
but A is a precondition for trusting any observation that would
validate B.

   inactive," and "55 orphaned entries" (actually 55 across two

   claims → Worker doesn't resolve → claims accumulate until
   purged. Every session without fixing this is a session that
   generates more queue debt. Not hypothetical — directly observed
   twice this session.

3. **The daemon reloading Python modules is not a built-in behaviour.**
   Today the gap was 4 hours of running stale code (pre-ADR-007
   sensor after post-reboot auto-start, no restart between commit
   and observation). Any future `src/` change that touches a
   daemon-imported module silently has zero effect until the
   daemon is restarted. `core-admin dev sync` syncs DB/vectors,
   not `sys.modules`. Worth formalizing as workflow: "any change
   to `src/will/`, `src/body/`, `src/mind/` requires
   `systemctl --user restart core-daemon` to reach the running
   daemon." A separate concern: not even logged anywhere that this
   staleness could happen.
