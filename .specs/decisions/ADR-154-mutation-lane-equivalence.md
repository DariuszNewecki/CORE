# ADR-154 ŌĆö Mutation-lane equivalence: retire `RemediationCeremony(write=True)` onto the ADR-109 human-gated proposal path

**Date:** 2026-07-19
**Governing paper:** none directly ŌĆö reconciles `RemediationCeremony` write-mode with the ADR-148 consequence chain and the ADR-109 human gate, the reconciliation ADR-153 explicitly left open
**Status:** Accepted
**Author:** Darek (Dariusz Newecki)
**Drafter:** Claude (session 2026-07-19 ŌĆö drafted under governor direction; surfaced by the 2026-07-19 external architecture review, ┬¦3b/┬¦4e/┬¦9d)
**Relates to:** ADR-109 (assisted-remediation lane ŌĆö the human-gated proposal form this ADR reuses), ADR-141 (assisted-lane validation extended to graph-independent audit-engine changes), ADR-148 (finalization barrier ŌĆö the standard the second lane must meet), ADR-101 (commit authorship), ADR-153 (ceremony extraction ŌĆö extracted, did not reconcile write-mode), ADR-068 D5 (`risk_classification.safe_auto_approval` authority), ADR-038 (circuit breaker on repeated proposal failures ŌĆö D3 execution-failure disposition), #799 (Production-Readiness epic ŌĆö implementation tracking)

## Context

CORE has **two** paths that apply AI-generated bytes to the production tree, held to unequal standards.

### Lane 1 ŌĆö mapped proposal path (strong; ADR-148)

```
finding ŌåÆ proposal (APPROVED) ŌåÆ claim.proposal (guarded CAS) ŌåÆ SandboxLifecycle
  hermetic worktree ŌåÆ declared-production propagation ŌåÆ CommitOutcome ŌåÆ FINALIZING
    ŌåÆ consequence_recorded_at ŌåÆ COMPLETED
```

Rollback is `ProposalExecutor.rollback_proposal`. `COMPLETED` is an *evidence* state (ADR-148): commit landed **and** consequence chain (pre_sha/post_sha/changed_files/resolved findings) durably recorded; reaper-reconstructable. Crucially, this lane may **auto-approve**: a proposal `compute_risk()` classifies as safe promotes under `risk_classification.safe_auto_approval` (`principal.system`, ADR-068 D5) with no human review. What licenses that autonomous authority is not a mapping alone but the full package ŌĆö **a deterministic mapping to a governed action or flow, combined with its declared safe risk classification and validation contract**; the change class is established-safe in advance.

### Lane 1b ŌĆö assisted lane (also strong, and human-gated; ADR-109/ADR-141)

The assisted lane already solves "an LLM-authored diff must reach production defensibly": an agent-authored, `assisted.validate_diff`-cleared patch becomes a **human-gated `DRAFT` proposal** via `LaneService.propose_validated_diff` (`src/will/autonomy/lane_service.py:171`). ADR-109 makes `approval_required=true` **mandatory** ("the governor reviews the actual validated diff before execution", ADR-109 line 112); approval runs `assisted.apply_diff` under `ProposalExecutor`. There is **no auto-approval** anywhere in this lane. ADR-141 only extended *validation* coverage to graph-independent audit-engine changes; it did not introduce autonomous authorisation.

### Lane 2 ŌĆö unmapped remediation ceremony (`RemediationCeremony(write=True)`) ŌĆö the outlier

```
finding (rule with no RemediationMap / no atomic fix action) ŌåÆ LLM crate ŌåÆ Canary
  ŌåÆ CrateProcessingService.apply_and_finalize_crate (live tree)
    ŌåÆ git_service.commit_paths([file], ŌĆ”) (direct commit) ŌåÆ mark findings resolved
```

Source: `src/will/remediation/ceremony.py` `_execute_file` (apply `:423`, commit `:447`). On a post-apply commit failure the code **explicitly leaves the modified file in the working tree**, marks findings `abandoned`, requests manual intervention (`:451-467`). Visible failure ŌĆö but not *transactional* failure, and no human ever reviewed the concrete diff.

### Why Lane 2 exists (the force any decision must respect)

`ViolationExecutorWorker` releases any claimed finding whose rule has since acquired a RemediationMap entry (`violation_executor.py:301`); what remains is findings for rules with **no deterministic atomic fix action**. The ceremony is the LLM-driven fallback for exactly those rules. This is precisely where **auto-approval is *least* defensible**: no deterministic mapping establishes that this class of change is safe, and Canary proves a candidate passes selected checks ŌĆö not that its architectural judgement is acceptable.

### Reachability today

`ViolationExecutorWorker.__init__(write: bool = False)` ŌĆö the daemon-live path defaults to dry-run. Write-mode is reachable through the `core-admin workers remediate` CLI in **both** rule-mode (`_run_rule_pipeline` ŌåÆ `ViolationRemediator(write=write)`, `remediate.py:316`) and file-mode (`_run_file_pipeline` ŌåÆ `RemediationCeremony(write=write)`, `remediate.py:204`); it is not currently daemon-scheduled. The worker declaration nonetheless permits daemon write-mode if the default is flipped.

### What ADR-153 settled and left open

ADR-153 extracted the ceremony from a Worker-to-Worker import and fixed blackboard attribution. It deliberately did **not** reconcile `write=True` with the consequence chain. This ADR closes that gap ŌĆö and, unlike Lane 1b, Lane 2 never adopted the human gate ADR-109 already made mandatory for LLM-authored production diffs.

### The three lanes are not the same ŌĆö and only Lane 2 is deficient

The two healthy lanes differ in *approval authority*, and that difference is legitimate. Lane 1 may auto-approve because a deterministic mapping (plus its risk classification and validation contract) establishes the change class as safe; Lane 1b mandates human review of the concrete diff because it carries no such deterministic authority. Lane 2 has neither the deterministic authority of Lane 1 nor the human gate of Lane 1b.

| Guarantee | Lane 1: mapped autonomous | Lane 1b: assisted | Lane 2 today (`write=True`) |
|---|---|---|---|
| Governed approval state | Ō£ģ | Ō£ģ | ŌØī |
| Human review of concrete diff | Not necessarily (`safe_auto_approval`) | Ō£ģ mandatory | ŌØī |
| Deterministic remediation mapping | Ō£ģ | ŌØī | ŌØī |
| Sandbox + declared-production propagation | Ō£ģ | Ō£ģ | ŌØī applies to live tree |
| Rollback on failure | Ō£ģ `rollback_proposal` | Ō£ģ | ŌØī apply ok, commit fails ŌåÆ uncommitted bytes on disk |
| ADR-148 consequence chain / proof state | Ō£ģ | Ō£ģ | ŌØī findings resolved on commit alone |

The commit is correctly scoped (`commit_paths([file_path], ŌĆ”)`, honouring ADR-101 D2). The gap is not authorship. Because Lane 2 is **unmapped**, it lacks the deterministic authority that would license autonomous approval ŌĆö so it belongs with Lane 1b: it needs the **human gate** (approval of the concrete validated diff) plus the isolation/rollback/evidence envelope both healthy lanes carry.

### Why this is application of ADR-109, not new policy

1. **ADR-109 already decided the principle ŌĆö for the lane Lane 2 belongs to.** ADR-109 does not require *every* AI-generated mutation to be human-approved; it requires that judgment-heavy, assisted changes *without deterministic autonomous authority* carry a mandatory human gate (`approval_required=true`, governor reviews the concrete diff). An **unmapped** rule lacks exactly that deterministic authority, so it falls squarely under ADR-109's gate. Lane 1's `safe_auto_approval` is untouched ŌĆö it is licensed by the deterministic mapping Lane 2 does not have.
2. **NorthStar: defensibility > productivity.** Two production-write lanes with unequal defensibility inverts the ranking; the cheaper lane is the weaker one.
3. **ADR-148.** `COMPLETED` is an evidence claim; Lane 2 produces no equivalent evidence, so its outputs are not independently reconstructable.
4. **Unmapped ŌćÆ higher scrutiny, not lower.** Absence of a deterministic mapping is an argument *for* the human gate, not against it.

## Decision

> `RemediationCeremony` becomes **candidate-only**. A candidate that passes crate/Canary validation **and** `assisted.validate_diff` may be converted into a **human-gated `DRAFT` proposal** containing the immutable validated patch and its declared production set. **No ceremony-generated bytes reach the production tree except through `ProposalExecutor`, `SandboxLifecycle`, explicit governor approval of the concrete candidate, and the ADR-148 finalization chain.** Direct ceremony application and direct ceremony commit are removed from the class.

Automatic **proposal creation** is reasonable. Automatic **proposal approval** is not supported by law and is explicitly out of scope (ADR-109 D3).

**Scope of this acceptance.** The equivalence *invariant* ŌĆö no AI-generated bytes reach the production tree outside the proposal chain, and ceremony write-mode is structurally removed (D4) ŌĆö is accepted now. Two items remain **deferred implementation decisions**, delegated to a later recorded governor decision, not open questions about the invariant: **D3a** (file-mode lineage) and **D5** (automatic draft submission vs discovery-only). No draft-creating path ships and ceremony write-mode stays disabled until both are resolved.

### D1 ŌĆö Reuse the ADR-109 proposal form; do not invent a new proposal type

The required execution shape already exists: `validated patch ŌåÆ assisted.apply_diff ŌåÆ DRAFT proposal ŌåÆ approval gate ŌåÆ ProposalExecutor`. `LaneService.propose_validated_diff(finding_id, patch, production_set)` already constructs exactly this (`DRAFT`, `approval_required=True`, `assisted.apply_diff` carrying the patch, finding deferred to the proposal). Implementation is therefore a *reuse/generalise*, not a new category. The candidate must clear the **same** validation gate the assisted lane's approval gate keys on ŌĆö Canary is not that gate:

```
LLM output ŌåÆ crate/Canary validation ŌåÆ extract exact patch
  ŌåÆ assisted.validate_diff (against validated_base_sha) ŌåÆ ValidatedRemediationCandidate
```

1. `RemediationCeremony` generates a candidate and runs its crate/Canary validation, then dispatches `assisted.validate_diff` on the extracted patch. Its terminus changes (see D3b) ŌĆö it no longer applies or commits.
2. Only on a cleared `assisted.validate_diff` does the trusted validation service construct the frozen `ValidatedRemediationCandidate` (D2) ŌĆö governance evidence is a first-class contract, never reconstructed ad hoc from crate internals, and callers may not hand-assert the verdict.
3. Reuse or generalise `propose_validated_diff` to consume that exact object (today it reads a single delegated finding via `fetch_delegated_finding`; the ceremony carries a *set* of claimed findings for one file ŌĆö see D3/D3a for the linkage/population reconciliation).
4. Link all source finding IDs to the created proposal.
5. Create the human-gated `DRAFT`.
6. `assisted.apply_diff` + `ProposalExecutor` own the production mutation; the ceremony owns nothing past candidate preparation.

A retained crate-application branch inside the ceremony would preserve exactly the parallel lane this ADR retires, so it is not kept.

### D2 ŌĆö Candidate immutability, as a first-class frozen contract

**The bytes validated, reviewed and approved must be exactly the bytes executed.** `propose_validated_diff` already freezes the *patch* and *production set* into the proposal action, but binds **no base SHA** ŌĆö a time-of-check/time-of-use gap between validation, approval and execution. Close it with an explicit contract rather than ad-hoc extraction. After crate/Canary validation **and** `assisted.validate_diff` succeed, the trusted validation service constructs and returns:

```
ValidatedRemediationCandidate {
  candidate_id
  patch
  patch_digest
  production_set
  validated_base_sha        # repo SHA the validation actually ran against
  validation_checks
  validation_results         # the cleared assisted.validate_diff verdict
  finding_ids
  rule_ids
  created_at
}
```

This makes the handoff auditable: *probabilistic generation ŌåÆ deterministic validated candidate ŌåÆ immutable proposal payload ŌåÆ human authorization ŌåÆ deterministic execution.* The proposal factory consumes this exact object.

**Construction is privileged.** `ValidatedRemediationCandidate` may only be constructed by the trusted validation service *after* `assisted.validate_diff` succeeds; `validation_results` is populated from that recorded run, never supplied by a caller. Otherwise a Canary result could be represented as though the assisted-lane validation gate had run when it had not ŌĆö the gate ADR-109 keys approval on.

**Fail-closed base-SHA behaviour (first implementation).** `assisted.apply_diff` MUST refuse execution unless repository `HEAD == validated_base_sha`. Any intervening commit requires candidate **revalidation and renewed approval** ŌĆö no building against an old base and propagating onto a newer tree. This may invalidate candidates after unrelated commits; that is the safe, unambiguous default. A later ADR may relax it using blob hashes for the production set and its relevant dependencies.

**Approval binds the tuple** `patch_digest + production_set + validated_base_sha`; changing any member invalidates the governor's approval.

This D-section is a prerequisite for the assisted lane generally, not only for ceremony findings ŌĆö closing it hardens Lane 1b too.

### D3 ŌĆö Finding lifecycle

The populations differ: ADR-109's lane items begin at `indeterminate+human`; `ViolationExecutorWorker` claims ordinary `open` findings. The reconciliation must be explicit rather than inherited:

```
open ŌåÆ claimed by ViolationExecutorWorker ŌåÆ candidate validated
     ŌåÆ deferred_to_proposal ŌåÆ DRAFT proposal
DRAFT approved + proposal COMPLETED ŌåÆ source findings resolved
DRAFT rejected                      ŌåÆ findings become indeterminate + human
execution failed                    ŌåÆ existing revival / circuit-breaker (ADR-038)
```

**Rejection disposition ŌĆö findings become `indeterminate + human`, not `open`.** A governor has explicitly rejected the concrete candidate; returning the findings to `open` would let the system rediscover them and regenerate essentially the same proposal indefinitely. Moving them to `indeterminate + human` is honest after an explicit rejection and reuses the assisted lane's existing human-delegation semantics rather than inventing a ceremony-specific retry loop. They return to `open` only on explicit governor action or a materially changed condition.

### D3a ŌĆö File-mode lineage (deferred implementation decision)

D1/D3 assume real claimed findings with durable blackboard IDs. That holds for worker and CLI rule-mode, but **not** CLI file-mode: it runs an ad-hoc audit, fabricates synthetic finding dicts (`_build_synthetic_findings`, `remediate.py:58`), uses `NullRemediationBlackboard` (`:76`), and creates no blackboard entries. Those synthetic UUIDs cannot be linked to or deferred by `propose_validated_diff`, which requires a live delegated finding.

Therefore, before file-mode may create a proposal, its audit findings must first be persisted as canonical blackboard findings with real IDs; only those persisted IDs may enter `ValidatedRemediationCandidate.finding_ids`. Synthetic UUIDs are diagnostic-only and must never serve as proposal lineage. **Alternatively** ŌĆö and this is a sub-decision for the governor ŌĆö file-mode remains **candidate-export-only** (emits the `ValidatedRemediationCandidate` for inspection, cannot auto-create a draft). Until this is resolved the ADR does not claim both modes follow the same lifecycle; rule-mode does, file-mode is gated on D3a.

### D3b ŌĆö Candidate-submission atomicity (no new split state)

The legacy dry-run terminus (post a review observation, mark source findings `abandoned`, return success) is a *discovery-only* behaviour and MUST NOT run when a draft is created. Draft creation requires a different terminal transition:

```
claimed findings ŌåÆ candidate validated ŌåÆ draft persisted ŌåÆ findings deferred_to_proposal
```

Candidate persistence, proposal creation, and the transition of **all** source findings from `claimed` to `deferred_to_proposal` form one logical transaction. A **Body-owned service performs candidate persistence, proposal creation, and finding deferral within one database transaction; Will coordinates the operation but does not implement compensating multi-service writes.** The findings must not first pass through `abandoned`. If the transaction fails, no partial draft/deferral state may remain ŌĆö claims are safely released or restored. This prevents the two new split states this change could otherwise introduce: *proposal exists but findings abandoned*, or *findings deferred but no proposal exists* ŌĆö and rules out an implementation that calls three individually valid services and attempts cleanup after the fact.

### D4 ŌĆö Remove write-mode structurally, across every authority surface (not just the class)

`write=True` must not merely become unused ŌĆö a future caller must be unable to reactivate the weak lane by passing it. Structural retirement covers *every* surface that carries the authority, code **and** declaration:

- remove `write` from `RemediationCeremony` (delete the `write` behavioural branch, `apply_and_finalize_crate`, and the direct `commit_paths` call);
- remove or replace the CLI `--write` option on `core-admin workers remediate`;
- remove `write` from the relevant worker constructors (`ViolationExecutorWorker`, `ViolationRemediator`);
- remove `crate.apply` and `git.commit` from the worker **declarations** ŌĆö a declaration is constitutional authority; leaving these after deleting the code retains dormant authority and misrepresents the worker's mandate;
- update declaration responsibility text from "apply to live `src/`" to candidate preparation;
- remove direct rollback-archive responsibilities that exist only for live application;
- update tests, CLI documentation, and the ADR-153 consumer descriptions;
- add a **standing rule** (cf. #798 proof-index ŌåÆ CI) that a candidate-only remediation module MUST NOT import live-apply or Git-commit services, plus a boundary test proving the ceremony **cannot** apply or commit.

The equivalence invariant ŌĆö *no path applies AI-generated bytes to the tree except proposal-backed, or is candidate/dry-run-only* ŌĆö is thereby enforced as an API-shape and declared-authority guarantee, not caller discipline.

### D5 ŌĆö The remaining posture dial (deferred implementation decision)

Existing law already answers "auto-approve vs human-approve" for *this* lane: an unmapped fix lacks deterministic authority, so it is human-gated (it does not qualify for Lane 1's `safe_auto_approval`). The open dial is narrower ŌĆö **whether draft submission is automatic**:

- **Option A ŌĆö automatic draft submission.** After Canary **and** `assisted.validate_diff` succeed, CORE automatically creates a human-gated `DRAFT` proposal containing the immutable validated candidate. The governor still reviews and approves the concrete diff before any write.
- **Option B ŌĆö discovery only.** CORE records the validated candidate; the governor (or an external agent) must explicitly submit it as a proposal. Candidate persistence must use a durable governed store keyed by `candidate_id`; an in-memory object, crate reference, or console rendering does not satisfy discovery-only retention.

In **both** options: direct ceremony application disappears; direct ceremony commit disappears; the actual production change requires approval of the concrete diff. The only difference is who presses "create draft." (A related sub-decision lives in D3a: whether file-mode persists findings to participate, or stays candidate-export-only.)

---

## Rejected alternative ŌĆö harden-in-place

Keep the direct commit, add a consequence row + pre/post sha so *some* evidence exists. **Rejected on its own terms:** consequence evidence alone does not provide pre-mutation human approval, hermetic isolation, rollback, or immutable candidate binding. Evidence parity is not lane equivalence; four of the five missing guarantees remain. (The external review's "don't add another guard inside `ProposalExecutor`" is corroborating evidence, not the basis of rejection.)

## Consequences

### Positive

- One production-write invariant: every AI-generated mutation requires governed authorization, sandbox isolation, rollback, commit attribution and durable consequence evidence. For **unmapped or judgment-dependent** fixes, authorization requires **explicit governor approval of the concrete validated diff** ŌĆö while Lane 1's mapped `safe_auto_approval` remains legitimate, licensed by its deterministic mapping.
- The "uncommitted bytes on disk after apply/commit split" failure mode disappears (SandboxLifecycle + `rollback_proposal` replace it).
- Candidate immutability (D2) hardens the *entire* assisted lane against TOCTOU, not just ceremony findings.
- Reuses ADR-109 machinery ŌĆö smaller and cleaner than a new proposal category with a retained crate-apply branch.

### Negative

- `propose_validated_diff` must generalise from single-finding to a finding-set for one file, and gain base-SHA binding (D2). Implementation tracked by #818 under epic #799.
- CLI file-mode needs a real lineage path (D3a) before it can create drafts, or must be scoped to candidate-export-only.
- More governor review load: unmapped-rule fixes now queue as drafts rather than self-applying. This is the intended trade (defensibility > productivity), not a regression.

### Neutral

- Candidate *generation and validation* are unchanged; the *terminus* changes (per D3b/D5): create a draft and defer findings, or record for submission ŌĆö never apply/commit. The legacy discovery side-effects (review observation + marking findings `abandoned`) survive only in discovery-only mode, not on the draft-creating path.

## Verification

Once implemented, the decision is demonstrable by the following standing checks (candidates for the D4 standing rule and its boundary tests):

- **No live-apply authority in the ceremony:** `RemediationCeremony` exposes no `write` parameter; `grep` finds no `apply_and_finalize_crate` or `git_service.commit_paths` reachable from `src/will/remediation/`.
- **No dormant declared authority:** the `violation_executor` / `violation_remediator` worker declarations carry neither `crate.apply` nor `git.commit`.
- **Import boundary:** a candidate-only remediation module importing a live-apply or Git-commit service is a rule violation (standing check, cf. #798).
- **Validation provenance:** a `ValidatedRemediationCandidate` cannot be constructed without a recorded, passing `assisted.validate_diff` run; a caller-supplied `validation_results` is rejected.
- **Base-SHA fail-closed:** `assisted.apply_diff` refuses when `HEAD != validated_base_sha` (fault-injection test: commit an unrelated change between validation and execution ŌåÆ execution refuses, requests revalidation).
- **Submission atomicity:** a fault injected into proposal creation leaves no findings in `deferred_to_proposal` and no orphan `DRAFT` (test asserts neither split state).
- **End-to-end:** an unmapped-rule finding produces a `DRAFT` proposal whose approval drives `ProposalExecutor` ŌåÆ `SandboxLifecycle` ŌåÆ commit ŌåÆ `FINALIZING` ŌåÆ `COMPLETED` with a durable consequence row.
