# ADR-119 — Scout: BYOR Path 1 induces a fitted constitution from source code

**Status:** Accepted — governor-ratified 2026-06-20
**Date:** 2026-06-20
**Grounding paper:** `CORE-BYOR.md` (§4 code×Induce cell) — primary.
**Amends:** ADR-108 D1 (rule defaults narrowed); ADR-111 D1 (project onboard scope narrowed)
**Builds on:** ADR-108 (machinery/rules split), ADR-111 (delivery mechanism),
              ADR-118 D2 (detect→suggest→confirm pattern)
**Closes:** T3 false path (bundling preset rules in wheel is wrong)

---

## Context

ADR-108 established the machinery/rules split and introduced `examples/starter-intent/` —
a small, authored starter carrying four universal rules: `symbol_ids` (blocking),
`docstrings`, `no_print`, `no_bare_except`. ADR-111 made `project onboard` deliver it
wholesale (machinery floor + four-rule constitution) into a target repo.

This was the correct artifact for the T1 verification task — proving the F-10 audit gate
works end-to-end against an external repo — but it embeds an assumption that breaks down
the moment the adopting repo is genuinely alien:

**Nobody knows what constitution is right for an alien repository until the code is read.
Not CORE. Not the governor. Not Claude.**

The four authored rules are universal enough to be *plausible* for any Python repo but they
are not *fitted*. A repo with strict type discipline needs different blocking rules than one
with 60% coverage. A library needs `no_print` blocking; a CLI tool's output layer should be
excluded from it. A codebase with endemic bare-except usage needs a migration ramp
(reporting first, blocking later), not an immediate FAIL gate on day one. A preset
constitution tells adopters what CORE's author believes is important — which may or may not
match what the adopter's code actually needs to govern.

ADR-111 D2 already banned the inverted form of this problem: KnowledgeGraphBuilder-style
synthesis that derives *what the code does* and calls it law. But it left no path for the
correct form — LLM-assisted *induction* of what the code *could be held to*, grounded in
the adopter's own patterns, using CORE's enforcement vocabulary as a reference grammar, with
mandatory human ratification before delivery. This ADR defines that path: **Scout**.

### CORE's own development cycle as an analogy

CORE is the only case where TARGET == SOURCE: CORE governs CORE. Every other BYOR path
governs an external target under a constitution the human writes for that target's specific
structure. CORE's `.intent/` was authored for CORE's source patterns — atomic actions,
blackboard, workers, Mind/Body/Will layers. Carrying any subset of CORE's rule-ids into an
alien repo is the same error as "one constitution governs all." The alien repo's author is
the one who should write its law, after CORE shows them what law *can* look like.

### GRC confirmed as BYOR Path 4

Before defining path codenames, a structural clarification: GRC (Counsel) IS BYOR Path 4.
`CORE-BYOR.md`'s three-row table omits it, but GRC shares the same Repository abstraction
(F-41/F-42/F-43) — the customer brings a document corpus (the Repository), CORE brings the
regulatory catalog (the Intent), and CORE evaluates compliance. The artifact type changes
(code → document corpus); the governance loop is structurally identical. This ADR names all
four paths.

## Decision

### D1 — BYOR path codenames

| Path | Codename | Repository type | Constitution source | Command |
|------|----------|-----------------|---------------------|---------|
| 1 | **Scout** | Code, no existing `.intent/` | Induced by Scout, ratified by human | `project onboard` + `project scout` |
| 2 | **Guard** | Code + existing `.intent/` | Human-authored; CORE enforces ongoing | `code audit --offline` |
| 3 | **Generate** | No artifact; has intent spec | Generated by CORE | `project new` |
| 4 | **Counsel** | Document corpus + regulatory catalog | CORE-provided catalog (GRC) | *(T5b — forthcoming)* |

Path 3 (Generate / `project new`) is a sibling of BYOR, not a BYOR path proper: it has no
pre-existing repository to govern. Named here to close the four-quadrant model.

### D2 — Scout is Path 1: definition and two-phase delivery

Scout's mission: given a code repository with no `.intent/`, induce a fitted constitution,
propose it to the human, and deliver it after ratification.

Scout delivers in two phases:

**Phase A — Machinery floor** (`project onboard`): Delivers the irreducible CORE substrate —
META schemas, taxonomies, enforcement/config — that every offline audit requires. Identical
for every repo. No rules. No LLM. This phase is already implemented by ADR-111; it continues
unchanged but with its scope narrowed (see D6 and the ADR-111 amendment): it delivers only
machinery, not the four-rule starter constitution.

**Phase B — Rule induction** (`project scout`): Reads the target repo's source code, uses
CORE's `.intent/` as a reference grammar (D4), produces candidate rules in CORE's enforcement
vocabulary, presents them for human ratification (D5), and writes the ratified set to
`<target>/.intent/rules/scout_inducted.json` plus companion mapping
`<target>/.intent/enforcement/mappings/scout.yaml`.

Phase A must precede Phase B — Scout's induction requires the machinery floor to be in place
so inducted rules are immediately auditable. An operator who wants to write their own rules
manually after Phase A is fully supported (Guard path with authored rules); Phase B is opt-in.

### D3 — Induction mechanism: detect → suggest → confirm

Scout's rule-induction engine follows the same three-stage pattern as `GRCApplicabilityGate`
(ADR-118 D2):

**1 — Detect.** Sample source files from the target repo (entry points, largest modules,
representative tests). Extract structural signals: naming patterns, import graph shape,
decorator usage, comment density, exception-handling style, print/logging patterns,
public-symbol coverage.

**2 — Suggest.** The LLM receives structural signals plus CORE's enforcement vocabulary —
engine names, `check_type` taxonomy, example rule JSON from `examples/starter-intent/` — as
reference. It proposes candidate rules: each candidate names a `rule_id`, maps to an existing
CORE engine (no new engine authoring in Scout), states a scope (path patterns), an enforcement
level (`blocking` / `reporting` / `advisory`), and a rationale derived from observed code
patterns.

The LLM is *not* told what to enforce. It is asked what the code's patterns suggest is
*enforceable*. If the code already follows a pattern universally, the candidate may be
`advisory` (make it visible) rather than `blocking` (it would never fire). If the code
violates a pattern widely, the candidate must note that enforcing it blocking would halt CI
on day one — the human calibrates the ramp.

**3 — Confirm.** The CLI presents each candidate to the operator: description, rationale,
sample evidence (file:line that would be flagged), proposed enforcement level. The operator
accepts, rejects, or adjusts each. Only the confirmed set is written to disk.

This is **not** the generation-from-code path ADR-111 D2 bans. ADR-111 D2 bans
KnowledgeGraphBuilder-style synthesis — deriving what the code *does* as a descriptor and
calling it law. Scout does the inverse: it observes what the code *could be held to*, proposes
it in CORE's enforcement vocabulary, and requires human ratification before it becomes law.
ADR-119 D3 is the canonical form of what ADR-111 D2 anticipated as "an explicitly-labelled,
non-authoritative suggestion for the human to consider."

### D4 — Reference grammar constraint: CORE's `.intent/` is vocabulary, not template

Scout uses CORE's `.intent/` as a *reference grammar* — a catalogue of what governance can
look like — not as a template to instantiate for the adopter.

- **In scope as reference**: CORE's enforcement vocabulary (engine names, `check_type` values,
  rule JSON schema, enforcement level taxonomy). Used to express inducted rules in CORE's
  standard grammar so they are immediately auditable by CORE's existing engines.
- **Out of scope**: CORE's own rule-ids, scope patterns (`src/body/**`, `src/mind/**`),
  policy references (`atomic_actions.*`, `architecture.*`), and constitutional structure
  (Mind/Body/Will layer concepts). None of these transfer to an alien repo.

An adopter's Scout-inducted constitution must be legible to an outsider who has never heard
of CORE's internal architecture. It is a statement of *that repo's* obligations, expressed
in a shared vocabulary — not an excerpt of CORE's obligations.

### D5 — Mandatory human ratification invariant

Scout MUST NOT write any rules to `<target>/.intent/` without explicit, per-rule operator
confirmation. This is not a UX preference — it is a constitutional invariant:

1. **Humans write the law** (UR-08). An LLM-inducted rule that is not ratified is a
   suggestion. A ratified rule is law. The moment of ratification is the human's authorship
   act.
2. **Defensibility.** If a rule blocks a commit, the human who ratified it owns the claim
   that this is correct. An auto-delivered, un-ratified rule has no owner and cannot be
   defended.
3. **Calibration.** The LLM may propose `blocking` for a pattern the human knows needs a
   ramp. Ratification is where calibration happens.

There is no `--accept-all` flag. `--write` writes the ratified set; without it the session
is a preview. The per-rule confirm step (D3 stage 3) is not skippable.

### D6 — Command model

| Operation | Command | Scope |
|-----------|---------|-------|
| Deliver machinery floor | `project onboard <target> [--write]` | Phase A — no LLM, no rules |
| Induce and ratify rules | `project scout <target> [--write]` | Phase B — LLM + ratification |
| Run offline audit | `code audit --offline` | Requires both phases complete |

`project onboard` scope is **narrowed from ADR-111 D1**: it no longer delivers the four-rule
starter constitution. Machinery floor only.

`project scout` is a new command. It refuses if `.intent/` machinery is absent (Phase A
incomplete). It reads the target repo, runs the induction loop (D3), and on confirmation
writes the ratified rules.

### D7 — LLM-unavailable fallback

If no LLM is available at `project scout` invocation time, Scout presents a curated menu of
universal, LLM-free candidate rules. The menu is exactly the four rules from
`examples/starter-intent/`: `symbol_ids` (blocking), `docstrings` (reporting), `no_print`
(reporting), `no_bare_except` (reporting). The ratification requirement (D5) still applies —
the operator accepts, rejects, or adjusts each.

This fallback makes Scout usable in airgapped or LLM-restricted environments. It is
explicitly the *fallback*, not the default. Its output is `examples/starter-intent/`'s rule
set — which is why that artifact is retained: it is Scout's LLM-fallback reference, not
`project onboard`'s default delivery.

### D8 — `examples/starter-intent/` revised role

`examples/starter-intent/.intent/` is retained with its current content. Its role narrows:

- **Illustrative** — shows a minimal, legible four-rule constitution in CORE's enforcement
  vocabulary. Referenced in docs; used as onboarding reading material.
- **Scout LLM-fallback** — the D7 fallback menu is exactly this rule set.
  `examples/starter-intent/` remains the governed source of truth for the fallback.
- **T1 regression fixture** — the enforcement-root test and consumer-mode verification
  (`tests/mind/governance/test_audit_context__enforcement_root.py`,
  `tests/mind/governance/test_stateless_audit__fails_closed.py`) remain pinned against it.

It is no longer the *default output* of `project onboard`. The four rules are one possible
outcome of Scout's confirm step if the human chooses them — not a preset.

### D9 — ADR-108 D3 scope correction: wheel bundles machinery floor only

ADR-108 D3 called for bundling the "starter" in the `core-runtime` wheel. Before this ADR,
"starter" meant machinery floor + four-rule constitution. After ADR-119, the bundled artifact
is **machinery floor only** — the ~27 CORE-substrate files (META, taxonomies,
enforcement/config). The rules layer is never bundled: it is per-repo-inducted or
per-repo-authored, with no canonical wheel-bundleable default. Issue #674 and its
implementation must be scoped to machinery floor only.

## Consequences

- **The on-ramp becomes epistemically honest.** CORE stops pretending it knows what
  constitution is right for an alien repo. Scout makes induction explicit and ratification
  mandatory.
- **`project onboard` loses its constitution delivery.** Operators who expected a four-rule
  constitution from `project onboard` must follow up with `project scout` (or author rules
  manually). The T2 docs (#640 step 2) must reflect the two-step model.
  `docs/cold-reviewer.md` and `docs/getting-started.md` need updating.
- **`examples/starter-intent/` is retained, its status changed.** It becomes illustrative +
  fallback, not the default artifact. The four rules are correct examples; they are not
  retired.
- **BYOR backlog update.** T1 done (enforcement verified 2026-06-20). T3 reframed: wheel
  bundles machinery floor only (D9). T3's scope is now clearly bounded.
- **Scout is a new command to implement.** Induction engine, CLI scaffolding, and ratification
  UX are all new. This ADR decides the shape; implementation is its own change-set.
- **No breaking change to Guard or Counsel.** Existing repos with hand-authored `.intent/`
  continue to use `code audit --offline` exactly as before. Counsel (GRC) is unaffected.

## Alternatives considered

- **Keep the four preset rules as `project onboard` default.** Rejected: no preset rule set
  is honest as a default for alien repos. The four rules may be miscalibrated for a specific
  codebase; delivering them without code analysis is a guess, and a guess is not a governance
  posture.
- **LLM-fallback menu only — no LLM induction.** Rejected: the four-rule menu is better than
  nothing but not fitted. LLM induction with ratification is the correct default when LLM is
  available.
- **Full auto-delivery without ratification.** Rejected on constitutional grounds (D5). UR-08
  is not negotiable.
- **Use CORE's `.intent/` as a prune-then-copy template.** Rejected: this is the ADR-108-
  forbidden copy, just smaller. CORE's vocabulary and CORE's rule-ids are different surfaces;
  the reference grammar constraint (D4) enforces this separation.
